# Future AI Extensions

本文档记录当前 MVP 暂不直接上线、但需要提前预留的能力。

## 1. 片尾剧情延展

目标：在短剧片尾暂停或留白画面处，让用户选择“下一步可能发生什么”，再由 AI 生成短剧情节预览。

建议分期：

1. V1：只生成文字版剧情预测卡，不生成视频。
2. V2：生成 3-5 张分镜图，配合字幕做轻量“预告片”。
3. V3：接入视频生成模型，生成 3-8 秒短视频。

必须注意：

- 生成内容需要打上“AI 猜测/非正片”标识。
- 不能误导用户以为是官方剧情。
- 生成结果要记录 prompt、模型版本、用户选择和审核状态。

## 2. 用户上传照片漫画化

目标：用户可上传头像，把片尾定格图漫画化，并生成“我也在剧里”的分享图。

建议分期：

1. V1：只支持本地预览，不上传云端。
2. V2：上传后生成漫画头像贴纸，不做真实换脸。
3. V3：在合规前提下做角色化合成。
4. V4：生成用户专属 AI 动画形象，支持“真实脸带入”和“动画形象带入”两种模式。

必须注意：

- 需要用户登录和明确授权。
- 用户原图、生成图、删除状态必须可追踪。
- 必须提供删除入口，避免隐私风险。
- 不做未成年人、公众人物、敏感身份的换脸扩展。

## 3. 用户系统预留

当前 MVP 暂不加入登录，但后续正常视频 App 必须支持用户维度。

建议表结构方向：

- `users`：用户基础信息和匿名设备迁移。
- `watch_sessions`：一次观看会话。
- `user_interactions`：用户对高光点的点击、选择、停留。
- `user_danmaku`：用户弹幕、审核状态、申诉状态。
- `user_assets`：用户上传图片、生成图、删除状态。

当前代码里 `session_id` 是临时匿名用户标识，后续登录后可以把匿名 session 合并到真实用户。

## 4. 好友开房与双人同看

目标：支持用户邀请好友进入同一个观看房间，共享播放进度、弹幕和互动结果。

建议分期：

1. V1：房间码加入，同步播放/暂停/进度。
2. V2：双人实时弹幕和表情互动。
3. V3：多人房间、好友关系、房间历史记录。

关键技术点：

- 需要登录体系或稳定匿名身份。
- 需要 WebSocket 或类似实时通道。
- 房主控制权、断线重连、延迟校准必须设计清楚。

## 5. 登录后的选片页面

目标：登录后根据用户历史、互动偏好和观看进度提供更完整的选片入口。

建议分期：

1. V1：最近观看、全部短剧、题材筛选。
2. V2：收藏、继续观看、互动热度榜。
3. V3：个性化推荐和运营位。

## 6. Web 到移动客户端

目标：MVP 先用 Web 移动端跑通，后续迁移到安卓或 iOS，提供公网访问能力。

建议选择：

- 演示和比赛优先：Web 移动端 + 公网部署。
- 客户端优先：Android 更容易落地和分发测试。
- iOS 更适合最终质感，但证书、审核和真机分发成本更高。

## 7. 语音与智能人控制

目标：播放和互动结束后，出现预设动漫形象作为智能人，接收语音指令，完成换片、调音量、打开弹幕等操作。

建议分期：

1. V1：固定按钮唤起语音输入，识别简单命令。
2. V2：智能人形象引导用户选择下一集或切换模式。
3. V3：结合剧情上下文做自然语言控制和推荐。

关键限制：

- 浏览器语音权限需要用户主动授权。
- 语音识别结果需要命令白名单，不能直接执行自由文本。
- 移动端系统权限和 Web 权限逻辑不同，客户端化时需要重做适配。
